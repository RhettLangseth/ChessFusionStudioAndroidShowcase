# Licensing Options

This is not legal advice. It is a practical publishing note for this showcase repo.

## Best Default For This Repo

If your main goal is recruiter visibility and your main concern is code reuse or theft, the safest practical default is:
- do **not** add a permissive open-source license yet
- keep the repo public only when you are ready for it to be copied and forked as a public artifact
- add a prominent notice in the README or a separate notice file stating that no license is granted for reuse

That keeps default copyright protection intact while still allowing the repo to function as a portfolio artifact.

## What That Means

Without an open-source license:
- people can still view the repo publicly on GitHub
- GitHub users can still fork it under GitHub platform behavior
- but you are not granting general permission to use, modify, or distribute the code

## What To Avoid If Reuse Is A Concern

Do not use MIT, Apache-2.0, ISC, or similar permissive licenses if your current priority is limiting reuse. Those licenses explicitly allow broad reuse.

## If You Later Want To Open It Up

If you later decide that broader reuse is acceptable, add a standard recognized software license in the repo root:
- ISC for a short permissive license
- Apache-2.0 if you want a more explicit patent-oriented permissive license

## If You Want A Custom Restriction Model

If you want the repo to be source-available with unusual restrictions, do not invent custom legal text casually. Use a lawyer-reviewed approach or a well-understood source-available license after separate review.
